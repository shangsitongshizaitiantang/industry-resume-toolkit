# 开发 / 工程师 Patterns

涵盖:**后端 / 前端 / 全栈 / 移动端 / 算法 / 数据工程 / 测试 / DevOps / 安全**。这个赛道在 GitHub 上已经很卷,但**目标进国内大厂的中文程序员简历**仍然有可优化空间。

> ⚠️ **本 toolkit 主打 industry 简历**,程序员简历有大量专门工具(如 LaTeX 模板、ATS 优化器、JD 匹配器)。如果你是纯英文 / 全球求职的程序员,推荐用 `Paramchoudhary/ResumeSkills` 等 ATS-focused 工具。本 patterns 更适合**目标国内大厂、需要中英文双版本、想跟 HR/技术面试官双向沟通**的工程师。

---

## 风格关键词

**技术深度 / 业务理解 / 工程能力 / 量化产出 / 系统思考**

招聘官(技术面 + HR)看程序员简历最在意:**技术栈对不对路、有没有真做过事、能不能讲清楚、有没有业务影响**。

---

## 核心规则

### 规则 1:技术栈分层 + 标熟练度

技术栈不能堆 30+ 个,要分层 + 标清楚熟练度:

```
精通(>1 年生产环境项目,能独立设计 + 排查问题):
  - 后端示例:Java / Spring Boot / MySQL
  - 前端示例:React / TypeScript / Webpack
  - 算法示例:PyTorch / Python

熟练(完整项目经验,能独立完成需求):
  - eg Redis / RabbitMQ / Docker

掌握(基础使用,有 1-2 个学习项目):
  - eg Go / Kubernetes / Elasticsearch
```

**禁忌**:把"了解 / 听说过"也列进精通(面试一问就漏)。

### 规则 2:每段经历要有"业务背景 + 技术方案 + 量化结果"

技术简历最容易写成"我用 X 做了 Y",但招聘官真正关心**为什么用 X、解决什么问题、效果多好**:

❌ "用 Redis 做缓存,提升了性能"
✅ "**针对商品详情页 QPS 峰值 5w+ 的场景**,**用 Redis Cluster 替代数据库直读**,**采用 Cache-Aside 模式 + LRU 淘汰策略**,**P99 响应时间从 320ms 降至 45ms,数据库 QPS 下降 80%**。"

业务背景 → 技术方案 → 量化结果,三段都不能缺。

### 规则 3:量化必须"工程级"

技术简历的量化指标:
- **性能**:QPS / TPS / P99 延迟 / 吞吐量 / 内存占用
- **规模**:数据量(GB/TB)/ 用户数 / 并发数 / 服务器数
- **效率**:开发周期缩短 / 部署速度 / 单元测试覆盖率
- **可靠**:可用性(几个 9)/ 故障率下降 / MTTR
- **业务**:业务指标提升(CTR / GMV / 留存等)

**示例**:
> "**主导支付系统改造**,**日峰值订单 50w → 200w**,**P99 从 800ms 降至 200ms**,**全年大促 0 故障**。"

### 规则 4:项目要分"个人贡献"vs"团队产出"

技术大厂面试官会专门追问"这是你做的还是团队做的"。简历里要分清:

> **项目背景**:[团队层面的业务]
> **团队规模**:[X 人,你的角色]
> **个人贡献**:**我负责 [Y 模块的具体设计/实现]**,**我独立完成 [Z 功能]**
> **协作部分**:与 [N 团队] 联调 [接口]

不能写"我做了 [整个系统]",这种话面试一问就漏。

### 规则 5:技术广度 + 深度兼顾

简历要让招聘官看到你**有专精领域 + 也有广度**:

- **深度**:1-2 个深度项目(性能优化 / 架构改造 / 复杂场景),写到原理 + 数据
- **广度**:涵盖 2-3 个不同领域(eg 后端 + 数据 + 中间件,前端 + 工程化 + 性能)

避免"全栈但都不深"或"只会一个框架"。

### 规则 6:开源 / 个人项目加分

国内大厂越来越看个人技术活跃度:
- **GitHub**(必须放链接,如果有可观项目)
- **开源贡献**(给知名 repo 提过 PR 是巨大加分)
- **技术博客**(掘金 / 知乎 / 个人博客)
- **技术分享**(团队/部门内部分享,有视频或 PPT)
- **Stack Overflow 高分回答**(海外加分)

**没有就不写**,有就突出。

### 规则 7:学校背景 + 算法竞赛

国内大厂(尤其后端/算法岗)看:
- **985/211** > 普通本科(尤其 0 经验校招)
- **算法竞赛**:ACM-ICPC / 蓝桥杯 / PAT / LeetCode 周赛排名
- **数学/计算机基础课成绩**

应届简历可以适度强调这部分;**社招 3+ 年后这些权重下降**。

### 规则 8:岗位细分 — 不同方向重点不同

不同细分方向的简历重点:

**后端**:
- 系统设计能力(高并发 / 分布式 / 微服务)
- 数据库 / 中间件深度(MySQL / Redis / Kafka / ES)
- 性能优化案例

**前端**:
- 工程化能力(Webpack / Vite / monorepo)
- 性能优化(首屏 / 渲染 / 包体积)
- 用户体验细节
- 跨端方案

**算法 / AI**:
- 模型选型 + 业务效果
- 论文 / 比赛(Kaggle / 天池)
- 数学 / 统计基础
- 大模型 / Agent 实战(2024 后必备)

**数据工程**:
- 数仓 / 数据治理 / ETL pipeline
- 大数据生态(Spark / Flink / Hadoop)
- 数据建模

**测试**:
- 自动化框架(Selenium / Appium / Postman)
- 测试策略 + 覆盖率
- 性能 / 安全 / 兼容测试

**DevOps / SRE**:
- 容器 / 编排(Docker / Kubernetes)
- CI/CD 工具链
- 监控 / 告警 / 容量规划

## 不同企业性质下的开发写法差异

| 企业性质 | 看重什么 | 写法重点 |
|---------|---------|---------|
| **大厂(BAT / 字节 / 美团等)** | 量化 + 系统设计 + 业务影响 | 高并发场景、性能优化、跨团队协作 |
| **创业公司 / 独角兽** | 全栈能力 + 快速迭代 | 0-1 从无到有、独立 owner、技术决策 |
| **外资科技(MS / Google / Apple)** | 英文 + 算法基础 + 工程深度 | STAR、Open-source 贡献、算法题表现 |
| **金融科技** | 严谨 + 高可用 + 合规 | 容灾设计、安全合规、压力测试 |
| **国企 / 政企** | 稳定 + 流程规范 | 项目经验完整、文档输出能力、合规意识 |

## 常见地雷

1. **技术栈堆叠** — 列 40 个工具但都浅尝辄止 → 5-8 个核心 + 熟练度
2. **没有量化** — "用 Redis 做了缓存" → 必须有 QPS / 延迟数据
3. **过度技术细节** — 简历贴源码截图 / 算法详解 → 招聘官没耐心看,放 GitHub 链接
4. **不分个人 / 团队** — 把团队成果说成个人,面试一问就漏
5. **业务无感** — 只讲技术不讲业务,面试官会问"你这个技术解决了什么业务问题"
6. **新技术堆叠** — 把听说过的 K8s / 大模型 / Rust 全堆上,但没真用过

## 校招 / 实习开发特别建议

- **GitHub 项目**比 GPA 重要(尤其有 Star / Fork 的)
- **算法竞赛**排名前列(eg LeetCode 周赛 Top 1k / ACM 区域赛奖)有效
- **开源贡献**给知名 repo 提过 PR(eg Vue / React / Spring / 国内大厂开源项目)
- **大厂实习** > 学校项目 > 自学项目(简历权重)
- **算法岗**:论文(顶会就业一线大厂直接 SP/SSP) + Kaggle 比赛
