## /加规则 — 把你发现的新规则加进 skill

用 toolkit 改自己简历的过程中,你可能会发现新的规则 / 判断 / 经验,想沉淀下来下次自动应用。这个命令把这些洞察按标准格式归位到对应文件。

(高级用户可选功能。普通用户不用这个命令也能正常用 toolkit。)

### 流程

1. **问用户**(一次一个,不要批量):

   **问题 1:这条新规则具体是什么?**(让用户用自己的话说清楚,可以举例)

   **问题 2:适用范围?**
   - (a) 通用 — 所有简历都适用 → 加到 `~/.claude/skills/industry-resume-toolkit/SKILL.md`
   - (b) 某个企业性质 — eg 只对互联网/科技大厂适用 → 加到 `patterns/企业性质/<类型>.md`
   - (c) 某个岗位 — eg 只对产品经理适用 → 加到 `patterns/岗位/<岗位>.md`
   - (d) 跨多个分类 — 同时加到多个文件

   **问题 3(只有选 a 通用时问):这条规则属于哪一层/章节?**
   - Phase 1 信息卫生
   - Phase 2 业务价值翻译
   - Phase 3 经历挖掘
   - §6 保真原则
   - 其他

2. **写规则之前先扫一遍目标文件**:
   - 看有没有已经存在的类似规则
   - 如果有,**不要新建一条**,而是在原条目下补充示例

3. **按标准格式写入**:

   对 `patterns/*.md`:
   ```markdown
   ### <一句话陈述规则>

   - **理由**:为什么这样
   - **示例**:(如果有具体改写)
   - **出处**:用户观察 YYYY-MM-DD
   - **适用条件**:(可选)
   ```

   对 `SKILL.md`:按对应章节的现有格式插入(列表项 / 编号 / 块引用),不要打破章节风格

4. **写完后简短汇报**:
   - 加到了哪个文件、哪个章节
   - 引用一下加进去的具体文字
   - 提醒用户:"下次跑 /改简历 时,新规则会自动生效"

### 重要

- 规则要写得具体、可执行。不要让用户写"要注重量化"这种空话,要追问"具体怎么量化、什么场景下用、有没有反例"
- 如果用户给的规则太模糊,**追问她要例子**,有了例子规则才有用
- 不要重复加已存在的规则
- 如果规则跟 SKILL.md 里的某条**冲突**了,停下来告诉用户:"这条跟 X 章节的 Y 规则可能冲突,你想替换、合并、还是保留两条?"
- 用第二人称称呼用户

### 备注

如果用户希望贡献这条规则到上游开源仓库(让其他用户也受益),提示她:
> "如果这条规则你觉得普适,欢迎在 GitHub 提 PR 贡献给主仓库,让其他人也用上。详见 [CONTRIBUTING.md](https://github.com/[your-username]/industry-resume-toolkit)"
